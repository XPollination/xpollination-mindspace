## MANAGEMENT ABSTRACT

**What:** Redesign the PM tool data model and dashboard to support deep recursive task trees where every node has an agent assignment, PDSA phase, and quality gates — replacing the current fixed-depth phases/items/sub_items structure.

**Why:**
- Thomas observes that our 3-agent process (human → orchestrator → PDSA → dev → quality gates) is itself a recursive task tree
- Tasks decompose into subtasks, which decompose further — forming a deep tree, not just depth 2
- Current v2 data model hardcodes 3 levels: `phases[]` → `items[]` → `sub_items[]`
- Current v2 dashboard renders those 3 levels with separate templates — cannot go deeper
- There is no visibility into WHO does WHAT at each level of the tree
- Quality gates exist only at the item level, not at every node
- The tree should be the "accurate tunnel" — showing from Thomas's strategic intent all the way down to the Dev agent's current operational task

**Outcome:** A PM tool where:
1. One recursive node type replaces phases/items/sub_items
2. Every node shows its owner (agent), PDSA phase, status, and quality gates
3. The tree can be arbitrarily deep — reflecting actual work decomposition
4. The scope stack is the path from root to the currently active leaf
5. Agent ownership is color-coded — instant visual "who does what"
6. This serves as the synchronization tool between Thomas and all agents

## PLAN

### 1. Process Reflection — Our Actual 3-Agent Flow as a Recursive Tree

Before designing the data model, let us map what ACTUALLY happens when Thomas gives a task. This is today's real process, observed during the Synology Chat webhook diagnosis (2026-02-01):

```
Thomas: "Diagnose the Synology Chat outgoing webhook failure"
│
├─ Orchestrator receives task, relays to PDSA
│   │
│   ├─ PDSA: Research & plan the diagnosis
│   │   ├─ PDSA: Analyze what we know (config, logs, network)
│   │   ├─ PDSA: Define diagnostic subtasks with acceptance criteria
│   │   └─ PDSA: Return plan to Orchestrator
│   │
│   ├─ Orchestrator: Quality-gate the plan (matches Thomas's request?)
│   │   ├─ YES → relay subtask to Dev
│   │   └─ NO → challenge PDSA, iterate
│   │
│   ├─ Dev: Execute diagnostic subtask 1 — Query Chat DB
│   │   ├─ Run: sudo psql -U postgres -d synochat
│   │   ├─ Extract: trigger_word, URL, token, channel_id
│   │   └─ Signal: "Done — config values extracted"
│   │
│   ├─ Orchestrator: Relay result to PDSA for analysis
│   │
│   ├─ Dev: Execute diagnostic subtask 2 — Test network path
│   │   ├─ Run: curl from NAS to Hetzner
│   │   └─ Signal: "Done — HTTP 200 received"
│   │
│   ├─ Dev: Execute diagnostic subtask 3 — Check synochatd logs
│   │   └─ Signal: "Done — libsynopersonalnotify.so.7 error found"
│   │
│   ├─ PDSA: Analyze all results, identify root cause
│   │   └─ Decision: "Trigger word colon mismatch"
│   │
│   ├─ PDSA: Update STUDY/ACT with findings
│   │
│   └─ Orchestrator: Report to Thomas
│       ├─ Thomas approves → COMPLETE
│       └─ Thomas provides feedback → next iteration
```

**Observation:** This is already **depth 5** (Thomas → Orchestrator → PDSA → Dev → specific command). And each node can spawn children — PDSA's analysis might discover a prerequisite (scope break), Dev's execution might hit a blocker that needs a new subtask.

**The key insight:** The depth is not fixed. It emerges from the actual complexity of the work. The data model must not constrain it.

### 3. v3 Data Model — The Recursive Node

#### 3.1 Core Principle: One Node Type

Everything is a **node**. What was a "phase" is now a node at depth 0. What was an "item" is a node at depth 1. What was a "sub_item" is a node at depth 2. And depth 3, 4, 5... are all possible.

```yaml
# v3 node schema (every node in the tree follows this)
node:
  id: string              # Unique identifier
  title: string           # Human-readable name
  owner: string           # Agent ID: "thomas" | "orchestrator" | "pdsa-agent" | "dev-agent"
  type: string            # "task" | "quality-gate" | "decision" | "scope-break" | "milestone"
  status: string          # "pending" | "active" | "paused" | "complete" | "failed" | "blocked"
  pdsa_phase: string|null # "plan" | "do" | "study" | "act" | "complete" | null
  pdsa_ref: string|null   # Reference to PDSA document (if this node IS a PDSA)
  description: string     # What this node represents (optional)
  quality_gates:           # Verification criteria (optional)
    - name: string
      status: string      # "pending" | "pass" | "fail"
      verifier: string    # Who checks: agent ID or "automated"
  dependencies: [string]  # IDs of nodes that must complete before this starts
  children: [node]        # RECURSIVE: same schema, any depth
```

#### 3.2 Agent Registry

Agents are defined once in `meta`, referenced by ID throughout the tree:

```yaml
meta:
  agents:
    thomas:
      display: "Thomas"
      color: "amber"        # Tailwind color family
      role: "Strategic direction, approval"
    orchestrator:
      display: "Orchestrator"
      color: "indigo"
      role: "Coordination, relay, quality gates"
    pdsa-agent:
      display: "PDSA"
      color: "violet"
      role: "Planning, research, quality review"
    dev-agent:
      display: "Dev"
      color: "emerald"
      role: "Implementation, commits"
```

This enables:
- Color-coded badges in the UI: `bg-amber-500/20 text-amber-400` for Thomas, etc.
- Agent legend/filter in the dashboard
- Future: agent-specific views ("show me only my work")

#### 3.3 Scope Stack

The scope stack is stored explicitly as an ordered array of node IDs, representing the path from root to the currently active deepest node:

```yaml
meta:
  scope_stack: ["root", "pm-tool", "pm-tool-v3-design"]
  active_node: "pm-tool-v3-design"
```

**Why explicit, not computed?** Because multiple children can be active simultaneously (parallel work streams). The scope stack represents where the HUMAN's attention is focused, which is a deliberate choice, not a derivation.

The scope stack drives:
- Breadcrumb rendering (same as v2, but now referencing tree nodes)
- Navigation: clicking a breadcrumb node collapses all other branches and expands the clicked path
- Context: when an agent reads the YAML, it knows where the human's focus is

#### 3.4 Status Propagation Rules

A node's status can be set manually OR derived from its children:

```
complete  : ALL children complete AND all quality gates pass
active    : at least one child is active (and no children failed)
paused    : was active, now suspended (scope break to work on something else)
blocked   : at least one dependency is not complete
failed    : at least one quality gate has status "fail"
pending   : not yet started (no active children)
```

Leaf nodes (no children) are set manually: pending → active → complete.

Non-leaf nodes derive their status bottom-up. This means when the Dev agent marks a subtask complete, the parent task's progress updates automatically.

#### 3.5 Node Type Semantics

| Type | Meaning | Visual |
|------|---------|--------|
| `task` | Work to be done. May have children (decomposition) or be a leaf (atomic work) | Default card style |
| `quality-gate` | A verification checkpoint. Has `quality_gates[]` with pass/fail criteria | Shield icon, colored border on pass/fail |
| `decision` | A point where a choice was made. `description` explains the decision | Diamond icon, info styling |
| `scope-break` | A child PDSA spawned because a prerequisite was discovered. Links to child PDSA | Lightning bolt icon, amber border (as in v2) |
| `milestone` | A significant achievement point, no work content itself | Flag icon, accent styling |

#### 5.2 Technical Approach — Flattened Tree Rendering

Alpine.js does not natively support recursive templates. The solution: **flatten the tree into a list with depth metadata**, then render as a flat `x-for` loop with CSS indentation.

```javascript
// Flatten tree into renderable list
function flattenTree(nodes, depth = 0) {
  let result = [];
  for (const node of nodes) {
    result.push({
      ...node,
      _depth: depth,
      _hasChildren: node.children && node.children.length > 0
    });
    if (node.children && !node._collapsed) {
      result = result.concat(flattenTree(node.children, depth + 1));
    }
  }
  return result;
}
```

Then render with depth-based indentation:

```html
<template x-for="node in flatNodes" :key="node.id">
  <div :style="'padding-left: ' + (node._depth * 24) + 'px'"
       class="flex items-center gap-2 py-2 border-b border-pm-border/30">

    <!-- Collapse toggle (if has children) -->
    <button x-show="node._hasChildren" @click="toggleCollapse(node.id)">
      <svg :class="node._collapsed ? '' : 'rotate-90'" ...>▶</svg>
    </button>

    <!-- Agent badge (color-coded) -->
    <span :class="getAgentClass(node.owner)"
          class="px-1.5 py-0.5 rounded text-xs font-medium shrink-0"
          x-text="getAgentDisplay(node.owner)">
    </span>

    <!-- Scope break icon -->
    <span x-show="node.type === 'scope-break'" class="text-amber-400">⚡</span>

    <!-- Title -->
    <span class="flex-1 truncate" x-text="node.title"></span>

    <template x-if="node.pdsa_phase">
      <!-- Same dot rendering as v2 -->
    </template>
  </div>
</template>
```

### 6. Migration Path — v2 to v3

The v2 `phases[]` → `items[]` → `sub_items[]` structure maps mechanically to the v3 `tree` structure:

```
v2 phase           →  v3 node at depth 0, type: "task"
v2 phase (scope_break) → v3 node at depth 0, type: "scope-break"
v2 item            →  v3 node at depth 1, child of its phase
v2 sub_item        →  v3 node at depth 2, child of its item
v2 quality_gates   →  v3 quality_gates on the same node (unchanged schema)
v2 pdsa_phase      →  v3 pdsa_phase on the same node (unchanged)
```

**Migration script** (Python or JS):
1. Read currentContext.yaml
2. For each phase: create a depth-0 node with `children = phase.items`
3. For each item: create a depth-1 node with `children = item.sub_items`
4. For each sub_item: create a leaf node
5. Wrap all depth-0 nodes in a root node
6. Write new tree structure to YAML

The dashboard code changes from three nested `x-for` loops to a single `x-for` over flattened nodes. This is simpler, not more complex.

### 7. What This Enables — The Synchronization Model

With v3, the PM tool becomes the **single source of truth** for the entire multi-agent workflow:

| Actor | Uses the tree for |
|-------|-------------------|
| **Thomas** | See the full tunnel from intent to execution. Know where every agent is. Approve quality gates. Set scope stack (attention focus). |
| **Orchestrator** | Read tree to know what's active. Relay tasks between agents by referencing node IDs. Verify quality gates before relaying results. |
| **PDSA Agent** | Read tree to know current scope. Create child nodes when decomposing work. Set quality gate criteria. Update PDSA phase. |
| **Dev Agent** | Read tree to know assigned subtasks. Mark nodes complete. Trigger quality gate verification. |

**The tree is the contract.** Agents don't need to remember what they were doing — they read the tree. Thomas doesn't need to ask "where are we?" — he sees the tree. The Orchestrator doesn't need to track state in memory — it's in the tree.

This is the "accurate tunnel" Thomas described: a transparent, persistent, shared view of the entire work decomposition from strategic intent to operational execution.
