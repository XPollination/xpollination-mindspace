## MANAGEMENT ABSTRACT

**What:** A formal framework for how PDCA cycles relate to each other hierarchically, how the project management tool must visualize this hierarchy, and how the MCP server will programmatically control quality gates across nested cycles.

**Why:** During implementation of the Project Management Tool (parent PDCA), we discovered that work items naturally spawn child PDCA cycles when scope breaks occur. The human and agent disconnect at these depth transitions. Without visibility into where we are in the tree, we lose alignment. Without programmatic quality gates, agents produce work that doesn't fit. This framework solves both.

**Outcome:**
1. A documented terminology and model for recursive PDCA cycles
2. Design specifications for the PM tool to visualize the PDCA tree
3. Design specifications for MCP server quality gate control
4. An operating protocol for human-agent collaboration at scope transitions

## PLAN

### 1. The Problem — Why We Need This

When the PDCA Agent (Claude) works with the Human (Thomas) on a project, a predictable pattern emerges:

```
1. PDCA-A is created to solve Problem X
2. During PLAN phase of PDCA-A, we discover Prerequisite Y
3. Prerequisite Y is out of scope for PDCA-A
4. We MUST stop PDCA-A and create PDCA-B for Prerequisite Y
5. PDCA-B itself discovers Prerequisite Z during its DO phase
6. PDCA-C is created for Z
7. Eventually Z is done → B can continue → A can continue
8. At each transition: Human and Agent lose sync
```

This is not a bug — it is the fundamental nature of complex work. Every established quality framework encounters this. The question is: how do we **make this visible and controlled** rather than chaotic?

### 2. What Exists — Thought Leaders and Established Frameworks

#### 2.1 Deming's PDSA Spirals

W. Edwards Deming described PDCA (he preferred PDSA — Study over Check) as **"spirals of increasing knowledge"**:

> "One can envision an open coil spring, with each loop being one cycle of the scientific method, and each complete cycle indicating an increase in our knowledge of the system under study."

Deming emphasized **Study over Check** because Check is binary (pass/fail) while Study is analytical (what did we learn? how does it change our theory?). This distinction matters: when a child PDCA completes, the parent should not just check pass/fail — it should **study** what was learned and potentially revise its plan for remaining children.

**Source:** Ronald Moen, "History of the PDSA Cycle," Deming Institute.

#### 2.2 Hoshin Kanri — Cascading PDCA

**Hoshin Kanri** (Policy Deployment), developed by Yoji Akao and practiced most notably by Toyota, is the most mature framework for cascading PDCA through organizational hierarchy:

- Strategic goals cascade down via the **catchball process** — a structured back-and-forth dialogue where each level catches goals from above, evaluates feasibility, and throws back feedback
- Each level creates its own **X-Matrix** mapping goals → objectives → activities → metrics
- Cascading X-Matrices form a tree: CEO's matrix spawns departmental matrices, which spawn team matrices
- Each level runs its own PDCA cycle

The **catchball process** is the closest existing mechanism to what we need at scope transitions: a formal protocol for negotiating between parent and child scope.

**Key limitation for our use case:** Hoshin Kanri assumes humans at every level. Our system has an agent operating autonomously between catchball moments.

#### 2.3 Boyd's OODA Loop Hierarchy

Colonel John Boyd's OODA loop (Observe-Orient-Decide-Act) was explicitly conceived as **multiple interacting loops at different organizational levels**:

> "The organizational hierarchy is polarized along the OODA loop. Senior people see and begin, junior people do and complete."

This maps directly to our pattern: **the Human observes and orients (what needs to happen), the Agent decides and acts (how to implement).** The disconnect at scope transitions is exactly the polarization Boyd identified.

Boyd's critical insight: effectiveness depends on minimizing the time the Human spends in the **Orient phase** at each scope transition — providing rich context so the Human can quickly re-orient.

**Source:** Chet Richards, "Boyd's OODA Loop — It's Not What You Think."

#### 2.4 Toyota Kata — Coaching as Nested PDCA

Mike Rother's **Toyota Kata** operationalizes Deming's ideas as a recursive coaching practice:

- The **Improvement Kata**: Understand Direction → Grasp Current Condition → Establish Next Target Condition → PDCA toward it
- The **Coaching Kata**: A coach guides a learner through the Improvement Kata at every level
- PDCA cycles within the Kata are **scientific experiments**: hypothesis → prediction → experiment → study delta

The coaching relationship maps to Human (coach) and Agent (learner). The disconnect happens when the learner encounters an obstacle requiring the coach to re-establish the target condition at a new depth level.

#### 2.5 Fractal Project Management

**Fractal planning** applies self-similarity across project scales:

> "It is a matter of seeing repeated patterns at all levels of the system and applying clear rules that remain consistent."

Key concepts:
- **Self-similarity:** The same PDCA pattern operates at every depth level
- **Fractal buffers:** Each level has its own variability absorber
- The WBS (Work Breakdown Structure) is inherently a recursive, self-similar decomposition

**Key limitation:** Existing fractal PM assumes the tree structure is known at planning time. Our system discovers the tree during execution.

#### 2.6 Agile Spikes — Discovering Hidden Work

The agile concept of **spikes** (from Extreme Programming) addresses the moment when execution reveals missing prerequisites:

> "A time-boxed investigation aimed at gaining knowledge or resolving uncertainties before work on a story can begin."

A spike reveals "stories hidden beneath the spike" — these are child work items in our tree. This is the "Do phase revealing need for children" in our framework.

### 3. Our Model — Recursive PDSA with Scope Transitions

Based on the established frameworks above, adjusted for human-agent collaboration:

#### 3.1 Terminology

| Term | Definition |
|------|------------|
| **PDSA Cycle** | A single Plan-Do-Study-Act cycle (using Deming's preferred "Study" over "Check") |
| **Root PDSA** | The top-level cycle that defines the ultimate goal |
| **Child PDSA** | A cycle spawned during the Do phase of a parent when a scope break occurs |
| **Scope Break** | The discovery during Do that a prerequisite must be built first |
| **Depth** | How many levels deep in the tree (Root = depth 0) |
| **Current Scope** | The PDSA cycle currently being actively worked on |
| **Scope Stack** | The path from Root to Current Scope (like a call stack) |
| **Scope Transition** | Moving between parent and child PDSA (down = scope break, up = child completed) |
| **Catchball** | The human-agent negotiation at a scope transition (from Hoshin Kanri) |
| **Quality Gate** | A programmatic, testable condition that must pass before a PDSA can be marked complete |
| **Completion Signal** | Agent declares "ready for testing" — triggers quality gate verification |
| **Study Delta** | The difference between expected and actual outcome at the Study phase |

#### 3.2 The PDSA Tree

```
Root PDSA: Agentic Development Framework
├── Child PDSA: Project Management Tool          ← SCOPE BREAK: "need visual alignment first"
│   ├── Child PDSA: Infrastructure Setup          ← depth 2, COMPLETED
│   │   ├── Hugo installation                     ← leaf item (no further PDSA needed)
│   │   ├── nginx config                          ← leaf item
│   │   ├── Site structure                        ← leaf item
│   │   ├── Tailwind CSS                          ← leaf item
│   │   ├── Alpine.js + SortableJS               ← leaf item
│   │   └── Live update testing                   ← leaf item
│   ├── Child PDSA: Recursive PDSA Framework      ← SCOPE BREAK: "need to understand the flow first"  ← WE ARE HERE
│   │   └── (this document)
│   ├── Child PDSA: Core Components               ← pending, depth 2

├── Child PDSA: Orchestrator Agent (Phase 2)      ← BLOCKED

```

#### 3.3 The Scope Stack (Current Position)

```
[0] Agentic Development Framework     → PAUSED (waiting for PM Tool)
[1] Project Management Tool            → IN PROGRESS (Phase A done, discovering Phase B needs)
[2] Recursive PDSA Framework           → ACTIVE (this PDCA) ← CURRENT SCOPE
```

The Scope Stack is the most important piece of context for both Human and Agent. It answers: **"Where are we? Why are we here? What do we return to when done?"**

```

                    │               PLAN                        │
                    │  • Define scope and quality gates          │
                    │  • CATCHBALL: Human ↔ Agent negotiate     │

                                   │
                    ┌──────────────▼───────────────────────────┐

                    │    → PAUSE this PDSA                      │

                    └──────────────┬───────────────────────────┘
                                   │

```

#### 3.5 The Three Layers

Our system operates at three distinct layers, each with different concerns:

| Layer | Name | Concern | Controlled By |
|-------|------|---------|---------------|
| **Layer 1** | Strategic | PDSA tree structure, scope stack, scope transitions | Human + PDCA Agent |
| **Layer 2** | Tactical | Individual PDSA execution (Plan, Do, Study, Act) | PDCA Agent + MCP Server |
| **Layer 3** | Operational | Implementation of leaf items within a PDSA's Do phase | Specialist Agents (Coder, Tester) |

```
Layer 1 — STRATEGIC: The Tree
───────────────────────────────────────────────────────
Human ←──catchball──→ PDCA Agent
"What needs to exist? What is blocked? What is the scope?"

Layer 2 — TACTICAL: The Cycle
───────────────────────────────────────────────────────
PDCA Agent ←──quality gates──→ MCP Server
"Is this PDSA on track? Are quality gates passing?"

Layer 3 — OPERATIONAL: The Work
───────────────────────────────────────────────────────
MCP Server ←──completion signals──→ Specialist Agents
"Implement this unit. Signal when done. MCP runs tests."
```

#### 3.6 Quality Gate Control Flow (Layer 2–3)

This is the programmatic loop Thomas described:

```
┌─────────────────────────────────────────────────────────────┐
│                    MCP SERVER (Controller)                    │
│                                                              │
│  1. Receives task assignment from PDCA Agent                │
│  2. Dispatches to Implementor Agent                         │
│  3. Implementor signals: "ready for testing"                │
│  4. MCP dispatches to Tester Agent                          │
│  5. Tester runs predefined quality gate tests               │
│  6. MCP evaluates results:                                  │
│     ├── PASS → marks task complete, notifies PDCA Agent     │
│     └── FAIL → returns to Implementor with feedback         │
│  7. Loop continues until PASS                               │
│                                                              │
│  ⚠️ NO CREATIVITY NEEDED — this is mechanical control       │
│  "this knitts everything together"                          │
└─────────────────────────────────────────────────────────────┘
```

### 5. The Pattern We Want to Establish

Thomas's guiding principle: **"Doing the right thing vs being busy."**

This means:
1. **Before any agent does work:** Verify we are in the correct scope (check scope stack)
2. **Before any agent starts implementing:** Verify the quality gates are defined (no gate = no work)
3. **After any agent claims completion:** MCP server verifies via tester agent (no self-grading)
4. **When a scope break occurs:** Full stop. Catchball with human. New child PDSA. Only then proceed.
5. **No creative interpretation:** The quality gates define done. The agent implements. The MCP verifies. No flexibility.

```
ANTI-PATTERN (Being Busy):
  Agent receives task → implements → says "done" → moves on
  ❌ No verification. No alignment. Busy but wrong.

PATTERN (Doing the Right Thing):
  PDCA Agent defines scope + gates → MCP dispatches to Implementor →
  Implementor signals "ready" → MCP dispatches to Tester →
  Tester runs gates → PASS/FAIL → Loop until PASS →
  PDCA Agent studies delta → Reports to Human
  ✅ Verified. Aligned. Right thing done right.
```
