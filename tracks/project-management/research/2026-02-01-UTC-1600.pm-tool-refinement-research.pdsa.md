## STUDY

### Problem 1: Source of Truth — Where Do PDSAs Live?

**The problem:** The test suite PDSA (`2026-02-01-UTC-1500.tmux-layout-test-suite.pdsa.md`) was created in `HomeAssistant/docs/pdca/`. The test script was committed to `HomeAssistant/tests/`. But the PM tool that tracks these tasks lives in `xpollination-mcp-server/docs/dashboard/currentContext.yaml`. There's no link from the PM tool task to the PDSA doc because they're in different repos.

**Why it happened:** The tmux layout work is infrastructure (HomeAssistant repo), but the PM tool tracking is in xpollination-mcp-server. The agent (me) followed the context I was in — HomeAssistant — and put the doc where it seemed natural.

**Analysis of options:**

| Option | Pros | Cons |
|--------|------|------|
| A. All PDSAs in xpollination-mcp-server | Single source of truth. PM tool `pdsa_ref` always resolves. | Infrastructure PDSAs feel out of place in an MCP server repo. |
| B. PDSAs stay per-repo, PM tool uses cross-repo refs | PDSAs live where the work happens. | `pdsa_ref` becomes `HomeAssistant:docs/pdca/filename.md` — fragile, not clickable. |
| C. PM tool repo is the "meta" repo for all work | Clean separation: PM tool tracks everything, repos hold code. | Duplication: PDSA knowledge exists in PM tool YAML AND in repo docs. |

**Recommendation: Option A with a convention.** All PDSA documents go to `xpollination-mcp-server/docs/pdca/` regardless of which system they target. The PDSA doc's metadata says which system it applies to. The PM tool `pdsa_ref` is always a local relative path.

Why: The PM tool is the orchestration layer. PDSAs are orchestration artifacts (plans, studies, learnings). Code and config stay in their respective repos. PDSAs are planning docs, not code — they belong with the planning tool.

Convention: PDSA filenames include the system scope: `2026-02-01-UTC-1500.homeassistant.tmux-layout-test-suite.pdsa.md`

### Problem 2: Naming — What Is a Group of Tasks?

**The problem:** The test suite (ts.1–ts.7) was tracked as a "phase" in `currentContext.yaml` (`id: "phase-test-suite"`). But "phase" in the original design meant a development stage (Infrastructure, Components, Interactivity, Visualization). The test suite is not a development phase — it's a focused unit of work.

**Analysis of terminology:**

| Term | Definition in established PM | Fits our usage? |
|------|------------------------------|-----------------|
| **Epic** | Large body of work, decomposed into stories (Agile) | Too Agile-specific. Thomas chose PDSA, not Scrum. |
| **Story** | A user-facing capability (Agile) | Does not apply — agents are the actors, not users. |
| **Phase** | A sequential stage of a project | Fits Phase A-D (sequential stages). Does NOT fit the test suite (parallel side-quest). |
| **PDSA Cycle** | A complete Plan-Do-Study-Act iteration | This IS what ts.1-ts.7 represents. It's a PDSA cycle with decomposed subtasks. |
| **Work Stream** | A parallel track of related work | Fits but too generic. |
| **Sprint** | Time-boxed iteration (Agile) | Doesn't apply — we don't time-box. |

**Recommendation: Use the v3 terminology already defined.**

In the v3 recursive node model, there is no "phase" or "epic" — there are only **nodes**. A group of tasks is simply a **parent node** with children. The type field distinguishes semantics:
- `type: "task"` — a unit of work (what was "phase" or "item")
- `type: "scope-break"` — a PDSA cycle spawned by discovery
- `type: "quality-gate"` — a verification checkpoint

The test suite (ts.1–ts.7) is a `type: "task"` node with 7 children. Its `pdsa_phase` field shows it follows PDSA methodology. No new terminology needed — the v3 model already solves this.

**One refinement:** Add `type: "pdsa-cycle"` for nodes that explicitly follow the full Plan→Do→Study→Act structure. This distinguishes "task groups that happen to have subtasks" from "deliberate PDSA cycles with quality gates between phases."

### Problem 3: Task Context — Self-Contained Task Descriptions

**The problem:** Early task descriptions (Phase A items like `a.1: "Install Hugo on Hetzner"`) are just titles. The test suite tasks (ts.1–ts.7) have rich descriptions with ROLE, CONTEXT, ACTION, ACCEPTANCE. The rich format worked — agents picked up tasks and executed autonomously.

**What worked in ts.1–ts.7 (pattern analysis):**

```yaml
description: >
  ROLE: dev-agent.
  CONTEXT: PDSA agent has defined test cases for tmux 3-pane layout verification.
    Test specs are in the PDSA document created in ts.1.
  ACTION: Implement the test cases as a runnable bash test script.
  ACCEPTANCE: Script runs without errors. Outputs clear pass/fail per test case.
```

This follows an **RCAA pattern** (Role, Context, Action, Acceptance) that maps directly to how an agent needs information:
1. **ROLE** — Am I the right agent for this? (prevents wrong agent executing)
2. **CONTEXT** — What do I need to know before starting? What inputs exist?
3. **ACTION** — What exactly do I do?
4. **ACCEPTANCE** — How do I know I'm done?

**Recommendation: Make RCAA the standard task schema.**

Add structured fields to the v3 node schema:

```yaml
node:
  # ...existing fields...
  role: string           # Agent ID that should execute this
  context: string        # What inputs/state exist from upstream tasks
  action: string         # What to do (imperative instructions)
  acceptance: string     # Definition of done (verifiable criteria)
```

This replaces the free-text `description` with structured fields. The description field can remain for human-readable summaries, but agents read the structured fields.

**Why not just use description?** Because the orchestrator constructs the relay message from these fields. With structured data, the orchestrator can generate: `"TASK FROM PDSA (relayed by Orchestrator): ROLE: dev-agent. CONTEXT: {context}. ACTION: {action}. ACCEPTANCE: {acceptance}."` — without having to parse free text.

### Problem 4: Chaining — Input/Output Contracts Between Tasks

**The problem:** `dependencies: ["ts.1"]` means "don't start ts.2 until ts.1 is done." But it doesn't express WHAT ts.2 receives from ts.1. Today, this was handled in prose: ts.2's CONTEXT says "PDSA agent has defined test cases in the document created in ts.1." But this is implicit.

**What actually flows between tasks (from today's observation):**

| From → To | What flows |
|-----------|------------|
| ts.1 → ts.2 | A PDSA document with test case specifications. File path needed. |
| ts.2 → ts.3 | A test script at a specific file path. |
| ts.3 → ts.4 | Test failure output (which tests failed, why). |
| ts.4 → ts.5 | Fix subtasks with acceptance criteria. |
| ts.5 → ts.6 | A fixed layout (state change, not a file). |
| ts.6 → ts.7 | Test results (pass/fail summary). |

**Three types of outputs flow between tasks:**
1. **File artifacts** — a document, script, or config at a specific path
2. **State changes** — the system is now in a different state (pane added, config reloaded)
3. **Analysis results** — findings, decisions, root causes (text)

**Recommendation: Add `inputs` and `outputs` fields to the node schema.**

```yaml
node:
  # ...existing fields...
  inputs:
    - from: "ts.1"                    # Source task ID
      type: "file"                    # file | state | analysis
      description: "PDSA doc with test case specifications"
      path: "docs/pdca/2026-02-01-UTC-1500.tmux-layout-test-suite.pdsa.md"
  outputs:
    - type: "file"
      description: "Executable test script"
      path: "tests/tmux-layout-test.sh"
```

This makes the chaining visible and machine-readable. When the orchestrator relays ts.2 to the dev agent, it can include: "INPUT: Read test specs from docs/pdca/2026-02-01-UTC-1500.tmux-layout-test-suite.pdsa.md (created by ts.1). OUTPUT EXPECTED: tests/tmux-layout-test.sh."

**Why this matters for agents:** An agent starting a task currently has to:
1. Read the task description
2. Guess what the previous task produced
3. Search for the relevant files

With explicit inputs, the agent knows immediately what to read and where to find it.

### Problem 5: Role Per Task — Confirmed Working Pattern

**The problem:** Thomas says this works well. Keep it.

**Evidence from today:** Every task in ts.1–ts.7 has an explicit `owner`. The orchestrator relayed tasks to the correct agent based on `owner`. When the orchestrator sent me (dev agent) ts.2, it included "ROLE: dev-agent" — I knew the task was mine. When the orchestrator sent ts.1 to the PDSA agent, it included "ROLE: pdsa-agent."

**Refinement:** The `owner` field in v3 already handles this. The proposed `role` field in the RCAA schema (Problem 3) reinforces it at the task level. Keep both — `owner` is the assignment, `role` in the description is the instruction.

### Synthesis: Refined v3 Node Schema

Combining all recommendations:

```yaml
node:
  id: string
  title: string
  owner: string              # Agent assignment (who executes)
  type: string               # task | pdsa-cycle | quality-gate | decision | scope-break | milestone
  status: string             # pending | active | paused | complete | failed | blocked
  pdsa_phase: string|null    # plan | do | study | act | complete | null
  pdsa_ref: string|null      # Relative path to PDSA doc (always in this repo)

  # RCAA — self-contained task instructions
  context: string|null       # What the agent needs to know before starting
  action: string|null        # What to do (imperative)
  acceptance: string|null    # How to verify done

  # Chaining — explicit input/output contracts
  inputs:                    # What this task receives from upstream
    - from: string           # Source task ID
      type: string           # file | state | analysis
      description: string
      path: string|null      # File path if applicable
  outputs:                   # What this task produces for downstream
    - type: string
      description: string
      path: string|null

  # Quality gates (unchanged from v3)
  quality_gates:
    - name: string
      status: string         # pending | pass | fail
      verifier: string

  dependencies: [string]     # Task IDs that must complete first (blocking)
  children: [node]           # Recursive subtasks
```

### What Changed vs. v3 Design

| Addition | Why |
|----------|-----|
| `type: "pdsa-cycle"` | Distinguishes deliberate PDSA cycles from task groups |
| `context`, `action`, `acceptance` | RCAA pattern — replaces free-text description for agent-executable tasks |
| `inputs[]`, `outputs[]` | Makes task chaining visible and machine-readable |
| Single-repo PDSA convention | All PDSAs in xpollination-mcp-server, scoped by filename |

### What Didn't Change

| Kept as-is | Why |
|------------|-----|
| Recursive node model | Already solved the depth problem |
| Agent registry in meta | Already solved the role visibility problem |
| Quality gates on any node | Already solved the verification problem |
| Scope stack | Already solved the "where are we" problem |
| `owner` field | Already solved role-per-task |

### Learnings

1. **Rich task descriptions are the difference between autonomous and dependent agents.** The ts.1-ts.7 tasks with ROLE/CONTEXT/ACTION/ACCEPTANCE ran nearly autonomously. Phase A tasks with bare titles required constant orchestrator guidance.

2. **Task chaining is an input/output contract, not just a dependency.** `depends_on: ts.1` means "wait for ts.1." `inputs: [{from: ts.1, path: "..."}]` means "read this file from ts.1." The second is actionable; the first is only a scheduling constraint.

3. **Single-repo for planning artifacts reduces friction.** When PDSAs are scattered across repos, the PM tool can't reference them with simple relative paths. Consolidating PDSAs to the orchestration repo (xpollination-mcp-server) keeps all planning artifacts navigable.

4. **The RCAA pattern (Role, Context, Action, Acceptance) is our equivalent of a user story format.** Agile has "As a [user], I want [thing], so that [value]." We have "ROLE: [agent]. CONTEXT: [state]. ACTION: [what]. ACCEPTANCE: [done criteria]." Both serve the same purpose: making work units self-contained and executable.
